7.1 Manajemen Proyek

Ringkasan singkat
Dokumen ini menjelaskan kebutuhan dan preferensi manajemen proyek untuk pengembangan dan pemeliharaan WA-Integrasi. Tujuan: menjaga komunikasi jelas, pengiriman bertahap, mitigasi risiko, dan keputusan terdokumentasi.

Metodologi yang direkomendasikan
- Pendekatan: Agile ringan (Scrum atau Kanban tergantung fase).
  - Fase pengembangan fitur baru: gunakan Sprint 2 minggu (planning, development, review, retrospective).
  - Fase pemeliharaan/ops: Kanban dengan WIP limit untuk tugas-tugas kecil dan hotfix.

Peran dan tanggung jawab
- Product Owner / Pemangku Kepentingan: menentukan prioritas fitur, menerima deliverable.
- Tech Lead / Maintainer: memimpin desain teknis, keputusan arsitektur, dan review PR.
- Developer: implementasi fitur, unit tests, dokumentasi perubahan.
- QA / Tester: verifikasi fungsional, acceptance testing; bisa digabungkan dengan dev untuk tim kecil.
- DevOps / Release manager: mengurus CI/CD, deployment, backup & restore.

Rencana komunikasi dan frekuensi
- Komunikasi asinkron (preferensi utama):
  - Issue tracker (GitHub Issues) untuk semua pekerjaan, bug, dan request; PRs untuk perubahan kode.
  - Review PR dan diskusi di issue → keputusan disimpan secara tertulis.
  - Notifikasi penting via chat (Slack/Teams/WhatsApp) untuk blocker/incident.
- Rapat sinkron (sinkronisasi singkat dan terjadwal):
  - Daily standup (opsional) 10–15 menit — hanya jika tim aktif bekerja dalam sprint.
  - Sprint planning (setiap 2 minggu) — 60–90 menit.
  - Sprint review / demo (akhir sprint) — 30–60 menit untuk menampilkan hasil ke PO.
  - Retrospective (akhir sprint) — 30–45 menit.
  - Weekly status (ops): laporan singkat mingguan (progress, blockers, risks).

Deliverables, acceptance & definisi selesai
- Deliverables: user story/issue selesai, PR merge, dokumentasi (README/CHANGELOG), migration scripts (jika ada), dan release notes.
- Acceptance criteria: tiap issue harus punya acceptance criteria yang jelas di deskripsi (expected behaviour, test data, success metrics).
- Definition of Done (DoD): kode teruji (unit/integration), lulus CI, code review 1–2 reviewer, dokumentasi diperbarui, dan verifikasi QA.

Alur kerja pengembangan & Git
- Branching: feature branches (`feature/<nama>`), hotfix (`hotfix/<nama>`), release (`release/<versi>`), main protected.
- Pull request: deskripsi jelas, langkah reproduksi, testing yang dijalankan, dan reviewer yang ditunjuk.
- Code review SLA: target 24–48 jam untuk response reviewer; eskalasi ke Tech Lead jika tertunda lebih lama.

Issue tracking, prioritas & estimasi
- Gunakan labels untuk prioritas (P0/P1/P2), jenis (bug/feature/task), dan area (backend/wa/frontend).
- Estimasi relative (story points) atau waktu (jam) untuk planning; gunakan estimasi konsisten antar sprint.

Testing & Quality
- CI: setiap PR harus memicu pipeline (lint, unit tests, basic integration tests).
- Coverage: minimal test pada core logic (recommended), acceptance test sebelum release.
- Manual QA: buat test checklist untuk fitur besar; sertakan test cases pada issue.

Release & Deployment
- Releases: buat tag release dan changelog. Siapkan rollback plan sederhana (backup DB, revert tag).
- Schedule deploy: hindari deploy di jam produksi sibuk; gunakan maintenance window jika ada perubahan besar.

Management of secrets & config
- Semua rahasia (API keys, DB passwords) harus di environment variables atau secret store; jangan commit ke repo.
- Enforce via CI secret-scan (recommended) dan gitignore untuk file/dirs sensitif (sessions, .status).

Escalation & incident handling
- Incident detection: alert via logging/monitoring (errors, high CPU/memory, Puppeteer crashes).
- Escalation: on-call person → Tech Lead → Stakeholder.
- Postmortem: ringkas root-cause, timeline, mitigasi; simpan di docs.

Preferensi komunikasi dan budaya kerja
- Preferensi: sedikit meeting, lebih banyak asynchronous updates; keputusan teknis dicatat di issue/PR.
- Respons SLA: blocker = immediate (chat + issue), normal PR/issue = 24–48 jam.

Change requests & scope control
- Semua change requests lewat issue RFC/feature request; besar kecilnya perubahan dikaji, diprioritaskan, dan disetujui oleh PO.

Risiko & mitigasi singkat
- Risiko: hardcoded secrets, long-running queries, Puppeteer instability, missing backups.
- Mitigasi: secret-scan, query timeouts, process supervisors (PM2), scheduled backups, health checks.

Penutup — ringkasan tindakan awal yang saya sarankan
1. Tentukan tool utama (Issue tracker + chat) dan SLA response singkat.
2. Terapkan DoD & pipeline CI minimal untuk PR.
3. Hapus fallback hardcoded API key dan tambahkan secret-scan CI.
4. Jadwalkan sprint/kerja ops berikutnya (atau kanban flow) dan set meeting cadence singkat.

Jika mau, saya bisa: (A) terapkan patch kecil untuk menghapus fallback API key, (B) buat template issue/sprint board sample, atau (C) mulai bab berikutnya.
